# 任务地基

## 目的

把模糊意图转换成后续 Codex 可以执行的地基文档，减少实现跑偏。优先明确边界、责任归属、契约、迭代计划、验证方式和用户偏好，再进入编码。

此技能用于规划和约束任务。除非用户明确要求继续实现，否则不要在地基阶段直接写代码。

## 必读资料

完成任何执行地基文档前，读取：

- `references/user-profile.md`：用户当前工作偏好。
- `references/grounding-template.md`：地基文档输出结构。
- `references/constraint-checklist.md`：缺失约束检查清单。

当任务会产出开发计划、交接文档、项目规则文件或实现说明时，读取 `references/user-agent-constitution.md`。除非用户给出更新的明确指令，否则把它视为用户本地规则来源。

当需要优化用户偏好档案，或用户询问为什么采用某种规划选择时，读取 `references/usage-log.local.md`。如果该文件不存在，先从 `references/usage-log.example.md` 创建。

## 工作流

1. 用一句话复述任务。
2. 判断任务类型：新项目、已有项目、功能、重构、排错、变现流程、系统集成、文档或审查。
3. 审计可用真源：用户需求、附件、仓库文件、当前 `AGENTS.md`、文档、脚本、测试和相关 git 状态。
4. 使用 `references/constraint-checklist.md` 建立约束矩阵。
5. 如果仍有阻断项，最多提出 3 个高价值问题。优先询问会决定责任归属、产品边界、数据契约、支付/安全、部署或验证的问题。不要先问审美类问题。
6. 如果可以用安全假设推进，说明假设并继续，将状态标为 `待确认`。
7. 先定义责任边界和模块契约，再写实现步骤。
8. 输出分阶段计划，每个阶段都要有验证门禁。
9. 当用户要求计划、交接或下一步文档时，生成或更新一份自包含 Markdown 地基文档。
10. 结尾给出一个推荐下一步，不给多个同级选项堆叠。

## 提问规则

- 只问会实质改变计划的问题。
- 问题要短、具体、可回答。
- 一轮最多问 3 个问题。
- 如果用户还在探索想法，主动给出默认建议。
- 如果用户说“先做计划”“先别写代码”“先划边界”等，保持规划模式。
- 如果用户问“下一步怎么做”，返回最小可验证下一步和停止条件。

## 准备度门槛

使用“95% 准备度”作为操作门槛，而不是数学保证。

只有满足以下条件，才可以标记为适合进入实现：

- 约束检查清单中的阻断项已经被约束，或明确标记为 `不适用`。
- 责任边界已经命名。
- 模块契约包含输入、输出、职责、不负责事项和验证点。
- 未知项已经分为 `阻断`、`可假设` 或 `可后置`。
- 第一阶段有明确文件或模块范围，并有验证命令或证据。
- 地基文档包含非目标和防跑偏规则。

如果门槛未通过，说明缺什么，并提出下一轮 1-3 个问题。

## 地基文档要求

使用 `references/grounding-template.md`。

地基文档必须让未来的 Codex 实例无需依赖聊天记忆也能继续执行。应包含：

- 目标与非目标。
- 假设。
- 产品边界。
- 用户角色与流程。
- 模块与责任边界图。
- 数据契约与状态流转。
- 适用时的 API / UI / CLI 契约。
- 适用时的支付、登录、权限、安全约束。
- 迭代计划。
- 验证门禁。
- 风险清单。
- 待确认问题。
- 给实现阶段的交接提示词。
- 来自 `references/user-profile.md` 的用户偏好规则。

只有在确实有帮助时才保存地基文档。默认位置：

- 已有仓库：`docs/task-grounding/<项目简称>.md`。
- 无项目或探索任务：`outputs/<项目简称>-grounding.md`。
- 如果用户给出路径，使用用户路径。

## 习惯迭代

每次有意义的使用结束时：

1. 在文件写入可用且用户没有禁止持久化时，向 `references/usage-log.local.md` 追加简短记录。优先运行 `scripts/record_usage.py` 保持格式一致。
2. 只有当用户明确表达稳定偏好、纠正技能行为，或同一偏好至少出现两次时，才更新 `references/user-profile.md`。
3. 永远不要记录密钥、私人 token、支付凭据、客户数据或生产敏感数据。
4. 偏好档案保持简洁，只记录会改变未来行为的规则。
5. 如果更新了偏好档案，在最终回复中说明。

示例：

```bash
python scripts/record_usage.py --task "U9/MES 对账工具规划" --goal "约束 MVP 和模块契约" --constraints "不碰生产库；先 Excel 导入导出" --assumptions "使用 Python + TypeScript" --next-adjustment "同类企业工具优先先做离线 MVP"
```

## 输出风格

- 使用简体中文。
- 简洁，但不省略边界、契约、验证和风险。
- 对比和约束覆盖优先用表格。
- 执行计划优先用编号阶段。
- 给出一条推荐主线。
- 地基未稳定前，避免过早展开实现细节。
