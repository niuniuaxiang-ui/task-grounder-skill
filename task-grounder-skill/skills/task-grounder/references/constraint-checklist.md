# 约束检查清单

用此清单检查任务是否已经足够落地。状态只允许：

- `已约束`
- `待确认`
- `可假设`
- `不适用`
- `阻断`

## 产品与范围

| 项 | 检查问题 |
|---|---|
| 目标 | 要解决的真实问题是什么？ |
| 非目标 | 本轮明确不做什么？ |
| 用户 | 谁使用，谁管理，谁付费，谁受影响？ |
| 场景 | 最高频、最高价值、最高风险场景分别是什么？ |
| MVP | 第一版最小可交付闭环是什么？ |
| 成功标准 | 什么证据证明完成？ |

## 架构与 owner

| 项 | 检查问题 |
|---|---|
| owner | 每个核心概念由哪个模块/层负责？ |
| 禁止 owner | 哪些层只能展示/接线，不能拥有业务真相？ |
| 模块边界 | 模块之间如何调用？ |
| 契约 | 输入、输出、错误、状态、幂等怎么定义？ |
| 复用 | 是否已有现成模块、库、组件或平台能力？ |

## 数据与状态

| 项 | 检查问题 |
|---|---|
| 数据源 | 真源在哪里？数据库、Excel、API、用户输入还是第三方回调？ |
| 数据模型 | 核心实体和字段是什么？ |
| 状态机 | 有哪些状态，谁能改变状态？ |
| 一致性 | 如何避免重复、丢失、乱序、金额/数量错误？ |
| 留痕 | 是否需要审计、日志、操作记录？ |
| 隐私 | 是否有个人、客户、生产敏感数据？ |

## 交互与接口

| 项 | 检查问题 |
|---|---|
| UI | 页面、空态、加载态、错误态、权限态是否定义？ |
| API | 路由、参数、返回、错误码、鉴权是否定义？ |
| CLI | 命令、参数、退出码、日志是否定义？ |
| 实时 | 是否需要 WebSocket、轮询、MQ、事件？ |
| 导入导出 | 文件格式、模板、错误报告、编码是否定义？ |

## 支付、账号与权限

| 项 | 检查问题 |
|---|---|
| 登录 | 是否需要账号、游客、管理员？ |
| 权限 | 谁能看、改、导出、审批、付费？ |
| 支付 | 是否需要官方支付回调、人工审核或沙箱？ |
| 解锁 | 支付/授权后解锁什么，多久有效？ |
| 安全 | 是否需要验签、防重放、限流、风控？ |

## 工程与交付

| 项 | 检查问题 |
|---|---|
| 技术栈 | 使用现有项目技术栈还是新建？ |
| 文件范围 | 第一阶段改哪些文件/目录？ |
| 依赖 | 是否新增依赖，为什么必须？ |
| 测试 | 单元、集成、浏览器、截图、接口、数据对账怎么验证？ |
| 部署 | 本地、内网、云服务器、桌面端还是移动端？ |
| 成本 | 域名、服务器、短信、支付手续费、AI API 是否预算？ |
| 运维 | 日志、备份、恢复、监控、告警是否需要？ |

## 风险

| 项 | 检查问题 |
|---|---|
| 阻断风险 | 现在不确认会导致返工或错误方向的是什么？ |
| 合规风险 | 是否涉及违法、侵权、支付违规、隐私泄露？ |
| 生产风险 | 是否会碰生产系统、生产数据或真实资金？ |
| 漂移风险 | 哪些需求容易越做越大，需要明确禁止？ |
